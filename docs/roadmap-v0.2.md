## 二、v0.2.0 主线

**评分驱动的个性化推荐**

v0.1.0 实现了评分偏好加权（局部影响），v0.2.0 围绕"评分数据"构建更完整的个性化链路：

> 用户评分越多 → 系统越懂你 → 推荐越准 → 用户越愿意评 → 良性循环

核心目标：
1. 让用户感知到"系统记住了我的口味"
2. 让低分反馈产生闭环（不只是降权，还能转化为行动建议）
3. 让推荐理由从"规则匹配"升级为"个性化解释"

## 四、v0.2.0 候选功能

### A. 用户偏好画像

**一句话**：基于评分历史自动提取用户的口味/类型/菜系偏好，在个人中心展示。

**用户价值**：
- 让用户看到"系统有多了解我"，建立信任
- 首次让评分行为产生可视化反馈（不只是隐性影响推荐）
- 为后续"手动调整偏好权重"打基础

**功能要点**：
- 统计用户评分过的记录，按 type_tags / cuisine_tags / taste_tags 聚合
- 显示"你喜欢辣 → 8 次高分 / 2 次低分 → 偏好度 80%"
- 显示"你对川菜似乎不太感冒 → 3 次低分"
- 纯展示，不做修改（v0.3 可加入手动调整）

**技术复杂度**：⭐⭐ 低
- 接口：1 个 GET 端点（`GET /api/v1/user/preference-profile`）
- 纯 SQL 聚合 + 内存统计，无新表
- 前端：profile 页新增一个卡片区域

**是否适合当前阶段**：✅ 非常适合
- 复用现有评分数据，零新数据结构
- 前后端工作量都小（后端 1 天 + 前端半天）
- 直接提升用户体验感知
- 面试可讲："通过用户行为数据自动构建偏好画像，无需用户手动填写"

---

### B. 推荐解释增强

**一句话**：让推荐理由从"适合午餐时段"升级为"因为这和你爱吃的 XX 很像"。

**用户价值**：
- 推荐可解释性大幅提升
- 用户理解"为什么推荐这个"，增加信任和采纳率
- 差异化体验：有评分历史的用户看到个性化理由，新用户看到通用理由

**功能要点**：
- 当评分偏好触发时，reasons 中明确写出命中的历史菜名：
  "和你给过 5 星的「剁椒鱼头」口味相似"
- 匿名/无评分用户显示："你是新用户，随机帮你挑一个～"
- 最近吃过展示具体天数："已经 5 天没吃凉皮了"
- 价位理由更友好："约 15-25 元，符合你的预算"

**技术复杂度**：⭐⭐ 低
- 纯文案改造，不改变打分逻辑
- `calculateScore` 中 reasons 构建逻辑扩展
- 需要 `ratedFoods` 中保留 food.name（当前仅保留 Food 对象，已包含 name）

**是否适合当前阶段**：✅ 非常适合
- 改动量极小（≈ 50 行 Java）
- 用户体验提升显著
- 和 A（偏好画像）搭配：画像展示宏观，理由展示微观

---

### C. 低分黑名单建议

**一句话**：当用户对同一分类/菜品连续低分时，主动建议"要不要把它加入黑名单或不想吃？"

**用户价值**：
- 闭环反馈：低分不只是降权，还能转化为行动
- 减少用户手动操作黑名单的心智负担
- 提升推荐满意度：不喜欢的东西越来越少出现

**功能要点**：
- 规则：同一 type_tag 或 cuisine_tag 累计 3 次 rating ≤ 2 → 触发建议
- 建议形式：推荐结果中追加 `suggestion: "你对「川菜」给了 3 次低分，要暂时不想吃川菜吗？"`
- 用户可一键操作（调已有 dislike API）
- 不强制，不弹窗打断推荐流程

**技术复杂度**：⭐⭐⭐ 中
- 需要新增规则引擎（累计低分检测）
- 需要新增 `suggestion` 字段到 RecommendResponse
- 前端需处理 suggestion 展示 + 一键操作
- 边界：避免重复建议骚扰用户

**是否适合当前阶段**：✅ 适合，但建议排在 A/B 之后
- 技术复杂度略高，需要前后端配合
- 涉及用户决策引导，需要仔细设计交互
- 优先级可略低于 A（画像）和 B（解释增强）

---

### D. 管理端反馈查看 UI

**一句话**：为 `POST /api/v1/admin/feedback` 提供简易 HTML 管理页面，替代 curl 操作。

**用户价值**（面向开发者/运营）：
- 不用查 API 文档就能查看反馈
- 一键切换反馈状态
- 降低反馈管理的使用门槛

**功能要点**：
- 纯前端 HTML 页面，放在 `admin/` 目录
- 列表展示：类型、评分、内容摘要、时间、状态
- 状态筛选：全部 / NEW / REVIEWED / RESOLVED / IGNORED
- 点击展开详情，支持状态切换
- 使用 `X-Admin-Token` 认证（localStorage 存储）

**技术复杂度**：⭐⭐⭐ 中
- 纯前端工程（HTML + vanilla JS），零新依赖
- 2 个现有 API 已就绪（查询 + 状态更新）
- 需要设计 UI（列表/详情/操作）
- 大约 1-2 天前端工作量

**是否适合当前阶段**：✅ 适合，但独立性高
- 完全不影响用户端功能
- 可以独立开发、独立部署
- 优先级取决于反馈数据量和运营需求
- 如果反馈量暂时不大，可延后到 v0.3

## 五、暂缓（不在 v0.2）

| 功能 | 原因 |
|------|------|
| 聚餐推荐（饭局投票） | 需求验证不足，单人推荐优先打磨 |
| 位置/餐厅/外卖 POI | 核心定位是"吃什么"不是"去哪吃"，v1.0 前不碰 |
| 小程序平台评价跳转 | 依赖外部平台接口，成本高收益低 |
| 菜品图片上传 | 当前 73 道菜无图片可先沿用占位图 |
| 社交分享推荐 | 有隐私顾虑，需谨慎设计 |
| 营养/热量信息 | 数据源不可靠，不可做 |

## 六、推荐实施顺序

### 阶段 1：用户感知层（第 1-2 周）

```
A（用户偏好画像）──→ B（推荐解释增强）
```

**理由**：两项都是低复杂度、高感知度的改进。A 展示"系统记住了什么"，B 展示"为什么推荐这个"。两者搭配形成完整的个性化叙事。

实施要点：
- A 需要在 profile 页新增一个"我的口味偏好"卡片
- B 只需要改 reasons 生成逻辑，前端无需改动
- 两项可并行开发（后端接口独立）

### 阶段 2：反馈闭环（第 3-4 周）

```
C（低分黑名单建议）
```

**理由**：依赖 A 完成后对评分数据的理解，且在 B 的推荐理由体系上增加 suggestion。此时用户已经通过 A/B 建立了"系统懂我"的认知，收到"要不要屏蔽川菜"的建议不会感到突兀。

实施要点：
- 后端新增累计低分检测逻辑
- RecommendResponse 新增 `suggestion` 字段（nullable）
- 前端推荐结果卡片底部展示 suggestion，支持一键操作

### 阶段 3：管理端补齐（可随时插入）

```
D（管理端反馈 UI）
```

**理由**：独立性强，与技术主线无依赖。可在阶段 1 或阶段 2 的间隙完成，也可根据反馈数据量决定是否提前。

实施要点：
- 新建 `admin/feedback.html`
- 纯前端页面，不依赖后端改动
- 使用现有 Admin API

## 七、v0.2.0 里程碑

```
v0.2.0 = A + B + C
```

| 指标 | v0.1.0 | v0.2.0 目标 |

## 八、不做事项（明确拒绝）

以下功能明确不在 v0.2.0 及之前版本范围内：

1. **不使用任何 AI/LLM 生成推荐或理由** — 保持纯规则，成本为零
2. **不接入第三方推荐服务** — 不依赖外部 API
3. **不收集用户位置数据** — 保护隐私，降低合规成本
4. **不做用户分层/A/B 测试** — 当前用户量不支撑统计显著性
5. **不做推荐效果指标面板** — 数据量不足，过度工程
