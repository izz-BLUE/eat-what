# 今天吃啥 - 推荐算法设计

## 概述

本项目采用**纯规则打分**推荐算法，不接入任何 AI 服务。

算法目标：
- 根据用户偏好推荐食物
- 降低最近吃过食物的推荐概率
- 避免推荐黑名单的食物
- 避免推荐有效期内不想吃的食物分类
- 从 Top 5 中随机选一个，避免结果固定

---

**已实现**：
- ✅ 获取候选食物池（enabled=1）
- ✅ 排除指定菜品（excludeFoodIds）
- ✅ 黑名单硬过滤（user_blacklist）
- ✅ 不想吃分类硬过滤（user_dislikes，有效期内）
- ✅ 用户选择分类硬过滤（typeTags + cuisineTags OR，最多3个）
- ✅ 餐段硬过滤（mealType 精确匹配）

- ✅ 最近吃过降权（eat_records，仅统计 EATEN 记录）
- ✅ 历史评分偏好加权（eat_records rating，正反馈加分/负反馈降权）
- ✅ 随机因素（0-19）
- ✅ Top 5 随机选一个

**后续阶段实现**：
- ⏳ 用户偏好权重（user_prefs）

## 算法流程

```
┌─────────────────────────────────────────────────────────────┐
│                    推荐算法主流程                            │
├─────────────────────────────────────────────────────────────┤
│  1. 获取候选食物池（enabled=1）                              │
│     ↓                                                        │
│  2. 排除指定菜品（excludeFoodIds）                           │
│     ↓                                                        │
│  3. 排除用户黑名单（user_blacklist）                         │
│     ↓                                                        │
│  4. 排除有效期内不想吃分类（user_dislikes）                  │
│     ↓                                                        │
│  5. 分类硬过滤（typeTags + cuisineTags OR）                  │
│     ↓                                                        │
│  6. 餐段硬过滤（mealType 精确匹配）                          │
│     ↓                                                        │
│  7. 口味硬过滤（清淡/辣/不辣）                               │
│     ↓                                                        │
│  8. 参考价位硬过滤（15以内→1, 15-25→2, 25-40→3, 40以上→4）  │
│     ↓                                                        │
│  9. 规则打分（最近吃过扣分 + 历史评分偏好 + 随机）            │
│     ↓                                                        │
│ 10. 按得分排序，取 Top 5                                     │
│     ↓                                                        │
│ 11. 从 Top 5 中随机选一个                                    │
│     ↓                                                        │
│ 12. 返回推荐结果                                             │
└─────────────────────────────────────────────────────────────┘
```

## 硬过滤规则

所有硬过滤在打分之前执行。任一硬过滤后无候选时，直接返回 null（Controller 返回 code=2002）。

### 分类硬过滤（第5步）

用户选择 typeTags 和 cuisineTags 采用 **OR** 逻辑：菜品只需命中任一个即可通过。

### 餐段硬过滤（第6步）

| 用户选择 | 过滤规则 |
|----------|----------|
| 不限/空 | 不限制 |
| 早餐 | food.mealTypes 必须包含"早餐" |
| 午餐 | food.mealTypes 必须包含"午餐" |
| 晚餐 | food.mealTypes 必须包含"晚餐" |
| 夜宵 | food.mealTypes 必须包含"夜宵" |

### 口味硬过滤（第7步）

| 用户选择 | 过滤规则 |
|----------|----------|
| 不限/空 | 不限制 |
| 清淡 | food.tasteTags 必须包含"清淡" |
| 辣 | food.tasteTags 必须包含"辣" |
| 不辣 | food.tasteTags 不包含"辣"也不包含"麻" |

### 参考价位硬过滤（第8步）

用户选择具体参考价位后，返回菜品的 priceLevel 必须与所选档位严格一致。

| 用户选择 | priceLevel | 说明 |
|----------|:--:|------|
| 不限/空 | — | 不限制价格 |
| 15以内 | 1 | 精确匹配，不含 2,3,4 |
| 15-25 | 2 | 精确匹配，不含 1,3,4 |
| 25-40 | 3 | 精确匹配，不含 1,2,4 |
| 40以上 | 4 | 精确匹配，不含 1,2,3 |

**重要约束**：
- 使用精确整数匹配，不允许范围兼容、软匹配或自动放宽
- 所选档位无候选时返回 null，由 Controller 返回业务码 2002
- recommend 与 swap 复用相同过滤逻辑
- "参考价位"是粗粒度估算，不表示真实门店售价

## 打分规则

### 总分公式

```
总分 = 最近吃过扣分 + 历史评分偏好加权 + 随机因素
```

分类命中、价格档位等已在硬过滤阶段保证，不再重复加分。
历史评分偏好加权仅对已登录且有评分记录的用户生效，匿名用户跳过。

### 最近吃过扣分

**数据来源**：`eat_records` 表，仅统计 `status = 'EATEN'` 的记录，使用 `MAX(eaten_at)` 获取最近一次。

**重要**：DECIDED（待用餐）记录不参与最近吃过降权。

| 距今时间 | 扣分 |
|----------|------|
| < 24h | -100 |
| 24h ~ < 48h | -80 |
| 48h ~ < 72h | -60 |
| 72h ~ ≤ 168h（7天） | -30 |
| > 168h | 0 |

### 历史评分偏好加权

**数据来源**：`eat_records` 表，统计当前用户所有 `status = 'EATEN'` 且 `rating IS NOT NULL` 的记录。

**启用条件**：
- 用户已登录（userId 非空）
- 用户有评分过的 EATEN 记录

**匿名用户不启用评分偏好加权**。

#### 正反馈（rating >= 4）

当前候选菜品与高分历史记录标签匹配时获得加分：

| 匹配维度 | 加分 | 说明 |
|----------|:----:|------|
| type_tags 有交集 | +8 | 同食物类型（如"火锅""快餐"） |
| cuisine_tags 有交集 | +8 | 同菜系/风格（如"川菜""日料"） |
| taste_tags 有交集 | +5 | 同口味（如"辣""清淡"） |

每维度仅检查是否有交集（任一标签命中即触发），不按命中标签数量累加。

#### 负反馈（rating <= 2）

当前候选菜品与低分历史记录标签匹配时降权：

| 匹配维度 | 降权 | 说明 |
|----------|:----:|------|
| type_tags 有交集 | -8 | 同类型 |
| cuisine_tags 有交集 | -8 | 同菜系 |
| taste_tags 有交集 | -5 | 同口味 |

#### 中性评分（rating = 3）

不参与偏好计算（不加分也不降权）。

#### 上下限

- **软上限 +20**：总个性化加分最高 +20
- **软下限 -20**：总个性化降权最低 -20
- 不做硬过滤（不会因为评分偏好排除任何菜品）

#### 推荐理由

- 正反馈加分 > 0 时，reasons 增加："符合你以往喜欢的口味/类型"
- 负反馈降权 < 0 时，不展示给用户（避免负面文案）
- 匿名用户不展示个性化理由

### 随机因素

加入 0-19 随机分，避免推荐结果固化。

## 从 Top 5 中随机选一个

**目的**：避免每次推荐结果固定，增加惊喜感。

```java
List<ScoredFood> top5 = scoredFoods.stream()
    .sorted((a, b) -> Integer.compare(b.score, a.score))
    .limit(5)
    .collect(Collectors.toList());

ScoredFood recommended = top5.get(new Random().nextInt(top5.size()));
```

## 返回推荐结果

**返回数据**：
```json
{
  "food": {
    "id": 1,
    "name": "猪脚饭",
    "category": "快餐",
    "typeTags": "快餐",
    "cuisineTags": "",
    "mealTypes": "午餐,晚餐",
    "tasteTags": "咸,香",
    "priceLevel": 2,
    "imageUrl": ""
  },
  "score": 5,
  "reasons": ["符合偏好分类", "符合参考价位", "最近几天没吃过，换换口味"]
```

**推荐理由**：
- 分类命中：`"属于「小吃」"`、`"菜系「日料」"`
- 餐段匹配：`"适合「午餐」时段"`

- 最近没吃过：`"最近几天没吃过，换换口味"`
- 评分偏好（如有正反馈）：`"符合你以往喜欢的口味/类型"`

## 特殊场景处理

### 场景 1：没有候选食物

**原因**：所有食物都被排除（黑名单、excludeFoodIds、不想吃分类、硬过滤无匹配）

**处理**：返回 null，Controller 返回 code=2002 `"当前条件没有合适菜品，请调整分类或口味"`

### 场景 2：userId 为空

**处理**：不查询黑名单、不查询不想吃记录、不查询最近吃过记录、不查询评分记录，只做基础推荐

### 场景 3：连续"换一个"超过候选数量

**处理**：当 excludeFoodIds 排除所有候选后，返回 null

### 场景 4：价格档位无候选

**处理**：所选参考价位档位无菜品时，返回 null（不自动放宽档位）

## 打分示例

### 场景描述

**用户参数**：
- mealType: 午餐
- priceLevel: 15-25（→ priceLevel=2 硬过滤）
- taste: 辣
- typeTags: 小吃
- userId: 1

**候选食物**（经过全部硬过滤后，priceLevel 已确保=2）：
| 食物 | typeTags | cuisineTags | tasteTags | 价格等级 |
|------|----------|-------------|-----------|:--:|
| 麻辣烫 | 小吃 | 川菜 | 辣,麻 | 2 |
| 凉皮 | 小吃 | 西北菜 | 辣,酸 | 2 |

**最近吃过**：麻辣烫（12 小时前）

**历史评分**（用户之前吃过的记录）：
- 火锅（typeTags=火锅, tasteTags=辣,麻, rating=5）→ 正反馈
- 钵钵鸡（typeTags=小吃, cuisineTags=川菜, tasteTags=辣, rating=2）→ 负反馈

### 打分过程

#### 麻辣烫
- 最近吃过：-100（< 24h）
- 评分偏好：
  - 正反馈 + 火锅 tasteTags "辣,麻" 交集：+5
  - 负反馈 + 钵钵鸡 typeTags "小吃" 交集：-8
  - 负反馈 + 钵钵鸡 cuisineTags "川菜" 交集：-8
  - 负反馈 + 钵钵鸡 tasteTags "辣" 交集：-5
  - 评分偏好合计：+5 - 8 - 8 - 5 = -16
- 随机：+10
- **总分：-106**

#### 凉皮
- 最近吃过：+0（未吃过）
- 评分偏好：
  - 正反馈 + 火锅 tasteTags "辣,麻" 交集：+5
  - 负反馈 + 钵钵鸡 typeTags "小吃" 交集：-8
  - 负反馈 + 钵钵鸡 cuisineTags "川菜" 交集：0（凉皮是"西北菜"）
  - 负反馈 + 钵钵鸡 tasteTags "辣" 交集：-5
  - 评分偏好合计：+5 - 8 - 5 = -8
- 随机：+15
- **总分：7**

### 最终结果

**Top 2**：凉皮(7)、麻辣烫(-106)。凉皮概率远高于麻辣烫。

**评分偏好效果**：虽然麻辣烫最近吃过被大幅扣分，但评分偏好进一步拉大了差距（-16 vs -8），让算法更倾向于推荐用户可能喜欢的食物。

### 示例 2：评分偏好上下限

**场景**：用户连续吃了 5 次火锅，每次都打 5 分。

**候选**：清汤火锅（typeTags=火锅, cuisineTags=川菜, tasteTags=鲜,清淡）

**历史评分**（5 条高评分火锅记录）：
- 麻辣火锅：typeTags=火锅, rating=5
- 鸳鸯火锅：typeTags=火锅, rating=5
- 牛油火锅：typeTags=火锅, rating=5
- 铜锅涮肉：typeTags=火锅, rating=5
- 菌汤火锅：typeTags=火锅, rating=5

每条记录 typeTags "火锅" 命中 +8，5 条 × 8 = +40 → **cap 到 +20**。

**总分**：0（未吃过）+ 20（评分偏好 cap）+ 5（随机）= **25**

即使只匹配 typeTags 维度，5 条高分记录就已触发 +20 上限。这保证了个性化偏好不会无限制放大，同时给予明显加分。
