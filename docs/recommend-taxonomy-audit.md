### 1.8 当前口味匹配/过滤规则

**硬过滤（matchesTaste）**：

| 用户选择 | 硬过滤规则 |
|----------|-----------|
| 不限/空 | 不限制 |
| 清淡 | 必须包含"清淡"，且不能包含"辣""微辣""麻" |
| 不辣 | 不能包含"辣""微辣""麻" |

| 重口 | 包含"辣""麻""酸"之一，或同时包含"咸"+"香" |

**软评分（calculateTasteScore）**：通过硬过滤后统一 +20。规则与硬过滤完全重合，导致口味维度退化为纯硬过滤，软评分失去区分度。

**额外的硬过滤（step 6）**：清淡时额外排除分类：火锅、烧烤、川菜、湘菜。

## 二、识别数据建模问题

### 2.1 category 混用不同层级（严重 🔴）

当前 `category` 字段同时承载两个维度：

| 维度 | 示例值 | 性质 |
|------|--------|------|
| **食物类型**（形态/做法） | 快餐、小吃、面食、火锅、烧烤、甜品 | 描述"这是什么形式的食物" |
| **菜系**（地域/文化） | 川菜、粤菜、湘菜、日料、西餐、家常菜 | 描述"这属于哪个菜系" |

**一个字段无法同时准确表达这两个维度。** 例如：
- "日料拉面"：食物类型是"面食"，菜系是"日料"，当前只能选其一（归类为"日料"）
- "麻辣烫"：食物类型是"小吃"，但无菜系属性
- "红烧肉"：通常配饭，菜系是"家常菜"

这种混用直接导致**餐段匹配错误**：把"日料"排除在午餐外，但"日料拉面"本质上是一碗面。

### 2.2 菜品分类不准确

| 菜品 | 当前 category | 问题 |
|------|-------------|------|
| 日料拉面 | 日料 | 本质是面食，午餐很常见 |
| 麻辣烫 | 小吃 | 通常作为正餐（午/晚），不是"小吃" |
| 炸鸡 | 快餐 | 也可能属于"小吃" |

### 2.3 缺少餐段语义

当前所有菜品都没有直接标注适用餐段。餐段匹配完全从 category 推导，导致：
- 甜品（奶茶）在任何餐段都不加分
- "家常菜"在晚餐不匹配
- 火锅在午餐不匹配

### 2.4 tasteTags 问题

#### 2.4.1 同义词未合并
- "辣" vs "微辣"——语义重叠，应合并为"辣"

#### 2.4.2 粒度不一致
- "咸"出现 14 次，几乎成默认值，区分度极低
- "麻"仅 2 次，无法独立支撑"麻辣"选项
- "微辣"仅 1 次，与"辣"几乎无法区分

#### 2.4.3 "重口"定义模糊
当前规则：辣/麻/酸之一，或咸+香同时出现。导致：
- "酸菜鱼"（酸,辣）→ 重口 ✅
- "糖醋排骨"（甜,酸）→ 重口 ✅（但糖醋排骨通常不被认为是"重口"）
- "猪脚饭"（咸,香）→ 重口 ✅（也不典型）
- 覆盖范围过大（约 20+ 道菜），几乎无区分度

#### 2.4.4 硬过滤与软评分规则完全重合
通过硬过滤的菜必然获得软评分 +20，使口味维度退化为纯硬过滤。

### 2.5 priceLevel 问题

1. **语义模糊**：文档说"价格等级"，又说"典型单人消费参考"。但前端直接展示"15元以内"，用户会理解为实际价格。
2. **分布不均**：level 4 仅 1 道菜（火锅）。
3. **"40元以上"选项缺失**：前端 VALID_PRICE_LEVELS 无此项，但 getPriceDisplay 有，逻辑不完整。

## 三、目标词典评估

### 3.1 餐段

**候选**：早餐、午餐、晚餐、夜宵

**评估**：
- ✅ 保留 4 个选项。
- ❌ 暂不增加"下午茶"：仅有 1 道甜品（奶茶），数据不足以支撑独立餐段。未来有足够甜品数据再增加。
- ✅ 餐段改为**硬过滤**：`food.mealTypes` 必须包含所选 mealType。

**最终**：早餐、午餐、晚餐、夜宵。全部硬过滤。

### 3.2 预算

**候选**：15元内、15-25元、25-40元、40元以上、不限

**评估**：
- ✅ 保留 5 档。增加"40元以上"（火锅 priceLevel=4 需要）。
- ✅ 保持软匹配，不加价档（无数据支撑）。
- ✅ 页面展示"参考价位"，明确实际价格因门店、份量和用餐方式而异。
- ❌ 不使用"经济实惠""豪华体验"等营销文案。

**最终**：5 档，软匹配，展示为"参考价位"。

### 3.3 口味

**候选评估**：

| 候选 | 判断 | 理由 |
|------|:---:|------|
| 不限 | ✅ 保留 | 默认行为 |
| 清淡 | ✅ 保留，改为硬过滤 | 7 道菜有此标签，必须包含"清淡"标签 |
| 辣 | ✅ 保留，含麻辣 | 7 道菜含"辣"，2 道同时含"麻"。当前仅 2 道"麻辣"不足以独立为筛选项 |
| 不辣 | ✅ 保留 | 排除"辣""麻"标签 |
| 重口 | ❌ 删除 | 定义模糊，覆盖过宽，区分度低 |

**关键决策**：
- **"辣"包含"麻辣"**：当前仅 2 道（麻辣烫、火锅），不独立。
- **"清淡"不再排除分类**：不按菜系排除火锅、烧烤、川菜、湘菜。仅按口味标签硬过滤。
- **三个口味选项全部硬过滤**：清淡必须含"清淡"、辣必须含"辣"、不辣必须不含"辣"和"麻"。不再有口味软评分。
- **旧缓存"重口"→"不限"**：两者不等价，"重口"不能迁移为"辣"。

**最终用户可选口味**：不限、清淡、辣（含麻辣）、不辣。共 4 个。

## 四、数据模型方案

### 方案 B（采纳）：扩展 foods 表字段

```sql
ALTER TABLE `foods`
  ADD COLUMN `type_tags` VARCHAR(128) DEFAULT '' COMMENT '食物类型，逗号分隔：快餐,小吃,面食,火锅,烧烤,甜品',
  ADD COLUMN `cuisine_tags` VARCHAR(128) DEFAULT '' COMMENT '菜系/风格，逗号分隔：家常菜,川菜,湘菜,粤菜,日料,西餐',
  ADD COLUMN `meal_types` VARCHAR(64) DEFAULT '' COMMENT '适用餐段，逗号分隔：早餐,午餐,晚餐,夜宵',
  MODIFY COLUMN `taste_tags` VARCHAR(128) DEFAULT '' COMMENT '规范口味标签，逗号分隔';
```

保留 `category` 字段作为**展示用单值**（向后兼容），新增字段为推荐逻辑使用。

**与初版方案的关键差异**：

| 项目 | 初版方案 | 修订方案 | 原因 |
|------|---------|---------|------|
| 食物类型字段 | `primary_category` 单值 | `type_tags` 多值 | 炸鸡、日料拉面可能同时具有多个类型 |
| 餐段 | 软评分 +30 | **硬过滤** | 选择早餐却仍可能推荐火锅，与此前口味错配属同一类问题 |
| 口味软评分 | +20 | **删除** | 通过硬过滤的菜统一加分无意义 |
| 无候选时 | 自动逐级放宽 | **返回 2002** | 用户明确选择的约束不应被后端静默忽略 |
| 清淡联动 | 排除火锅/烧烤/川菜/湘菜 | **取消** | 仅按口味标签硬过滤 |

## 六、推荐逻辑方案

### 6.1 过滤和评分顺序

```
① enabled=1（候选池入口）
  ↓ 硬过滤
② excludeFoodIds（换一个排除）
  ↓ 硬过滤
③ user_blacklist（黑名单排除）
  ↓ 硬过滤
④ user_dislikes 有效期内（不想吃分类排除，按 category 匹配）
  ↓ 硬过滤
⑤ 分类筛选（type_tags + cuisine_tags，OR，最多 3 个）
  ↓ 硬过滤
⑥ 餐段硬过滤（food.mealTypes 必须包含所选 mealType；未选则跳过）
  ↓ 硬过滤
⑦ 口味硬过滤：
    - 辣 → food.tasteTags 必须含"辣"
    - 不辣 → food.tasteTags 不能含"辣"或"麻"
    - 清淡 → food.tasteTags 必须含"清淡"
    - 不限 → 跳过
  ↓ 候选为空 → 返回 2002 "当前条件没有合适菜品，请调整分类或口味"
  ↓ 候选非空 → 软评分
⑧ 预算软匹配（+25）：food.priceLevel 匹配用户选择（"不限"则不加分）
⑨ 最近吃过降权（-100 ~ 0）
⑩ 随机因素（0-19）
  ↓
⑪ 按得分排序，取 Top 5
⑫ 从 Top 5 随机选一个
```

### 6.2 各阶段明确语义

| 阶段 | 类型 | 说明 |
|------|:---:|------|
| ① enabled=1 | 硬过滤 | 仅启用菜品 |
| ② excludeFoodIds | 硬过滤 | 换一个功能 |
| ③ blacklist | 硬过滤 | 用户明确拉黑 |
| ④ dislikes | 硬过滤 | 临时不想吃，有过期时间，按 category 匹配 |
| ⑤ 分类筛选 | 硬过滤 | type_tags + cuisine_tags，OR 关系，最多 3 个 |
| ⑥ 餐段 | **硬过滤** | food.mealTypes 必须包含所选 mealType |
| ⑦ 口味 | **硬过滤** | 辣/不辣/清淡各自独立规则，不再有软评分 |
| ⑧ 预算 | 软评分 +25 | 唯一保留的软评分维度 |
| ⑨ 最近吃过 | 扣分 | 维持现有规则（仅 EATEN 记录参与） |
| ⑩ 随机 | +0~19 | 维持现有规则 |
| ⑪ Top 5 | 截断 | 维持现有规则 |
| ⑫ 随机选一 | 输出 | 维持现有规则 |

### 6.3 关键设计决策

**Q: 哪些条件是硬过滤？**
- 黑名单、不想吃、分类筛选、餐段、口味 → 全部硬过滤
- 预算 → 唯一软评分维度

**Q: 多选类型和菜系采用 OR 还是 AND？**
- **OR**。选"快餐 + 川菜"意味着"我想吃快餐或者川菜"。

**Q: 同时选择类型和菜系时如何组合？**
- 取并集。例如选"面食"+"日料"，匹配条件为 `type_tags LIKE '%面食%' OR cuisine_tags LIKE '%日料%'`。总共不超过 3 个。

**Q: 无候选时是否自动放宽？**
- **不自动放宽。** 返回 2002 + 提示"当前条件没有合适菜品，请调整分类或口味"。
- 前端提供显式按钮"放宽条件重新推荐"，由用户主动触发。
- 放宽顺序：先去掉分类 → 再去掉口味 → 最后仅保留黑名单+不想吃+excludeFoodIds。

**Q: 是否保留 Top 5 随机？**
- ✅ 保留。

**Q: 推荐理由如何调整？**
- "符合预算" → 预算匹配（唯一加分理由）
- "最近几天没吃过，换换口味" → 最近吃过扣分为 0
- "符合偏好分类" → 分类硬过滤命中
- 删除"适合当前餐段"（硬过滤后全匹配）、"符合口味偏好"（硬过滤后全匹配）

**Q: 清淡是否仍自动排除重口味分类？**
- ❌ 不再联动。清淡仅按 tasteTags 包含"清淡"做硬过滤，不与 category/type_tags/cuisine_tags 耦合。

### 6.4 与旧算法对比

| 维度 | 旧算法 | 新算法 |
|------|--------|--------|
| 餐段 | 从 category 推导，软评分 +30 | 从 food.mealTypes 精确匹配，**硬过滤** |
| 口味"重口" | 硬过滤 + 软评分 +20 | **删除** |
| 口味"辣" | 硬过滤（含微辣）+ 软评分 +20 | **仅硬过滤**（含辣），无软评分 |
| 口味"不辣" | 硬过滤 + 软评分 +20 | **仅硬过滤**，无软评分 |
| 口味"清淡" | 硬过滤 + 软评分 +20 + 排除分类 | **仅硬过滤**（含清淡标签），不排除分类 |
| 预算 | 软评分 +20（缺40元以上） | 软评分 **+25**（含40元以上） |
| 分类 | 单值 category，最多 3 个 | type_tags + cuisine_tags 双维度 OR，最多 3 个 |
| 无候选 | 返回 null | 返回 **2002**，不自动放宽 |

## 九、产品决策确认（已定稿）

以下决策已在本次修订中全部纳入，不再需要额外确认：

| # | 决策项 | 结论 |
|:---:|------|------|
| 1 | 口味词典 | 删除"重口"，辣含麻辣，最终：不限、清淡、辣、不辣 |
| 2 | 口味全部硬过滤 | ✅ 清淡/辣/不辣全部硬过滤，删除口味软评分 |
| 3 | 清淡与分类联动 | 取消。清淡不再排除火锅/烧烤/川菜/湘菜分类 |
| 4 | 旧缓存"重口"迁移 | 迁移为"不限"，两者不等价 |
| 5 | 预算展示 | 保留 5 档具体金额 + "参考价位"提示，不用营销文案 |
| 6 | 预算"40元以上" | 增加。priceLevel 实际有 4 档（1-4），5 暂不使用 |
| 7 | 分类拆分 | type_tags（6 个）+ cuisine_tags（6 个），OR，最多 3 个 |
| 8 | 数据模型 | 方案 B，type_tags 为多值字段 |
| 9 | 餐段硬过滤 | food.mealTypes 必须包含所选 mealType |
| 10 | 禁止自动放宽 | 无候选 → 2002，显式按钮由用户主动触发 |
| 11 | 元数据接口 | `GET /api/v1/meta/recommend-options`，后端代码常量，不建表 |
| 12 | 日料拉面 | type_tags=面食, cuisine_tags=日料 |
| 13 | 麻辣烫 | meal_types 含午餐 |
| 14 | 火锅 | meal_types 含午餐 |
| 15 | 臭豆腐 | 不增加"臭"标签 |
| 16 | 奶茶 | meal_types 为空，暂不进入四餐段推荐 |
| 17 | 微辣 | 统一归一化为"辣" |
| 18 | Migration | 单一 V6，按菜名匹配，检查每条 UPDATE 影响行数 |

> **结论**：当前推荐体系在分类模型和口味标签方面存在结构性缺陷。方案 B（扩展 type_tags/cuisine_tags/meal_types 三个多值字段）+ 餐段口味全部硬过滤 + 预算唯一软评分，是从根本上解决问题的方案。5 个阶段，预估总工时 10-15h。单一 V6 Migration 原子完成字段新增和 30 道菜数据迁移。
